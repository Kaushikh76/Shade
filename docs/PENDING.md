## MPC — Production Gaps (current implementation is testnet-grade)

| Item | Detail |
|---|---|
| **Persistent committee keypairs** | ✅ Fixed — `keys.ts` persists Ed25519 keypairs in `mpc_committee_keys` DB table via `loadOrGenerateKeys()`. |
| **Real Merkle root in `mpc_settle`** | ✅ Fixed — `computeMpcRoot` Windows bug resolved (removed `existsSync` guard that failed on `.exe` suffix). Real lean-IMT root now computed via `coinutils compute-root` and passed to `mpc_settle` (`cdf4abe`). |
| **Output commitments are `recipient_commitment`** | Current DB stores `recipient_commitment` as the output. The proper derivation: for each matched intent, the counterparty's `note_commitment` becomes the output commitment (i.e. intent A's output = intent B's input note, and vice versa). Review and tighten this mapping. |
| **Real intent amounts via Shamir** | Integration test uses random nullifiers/commitments. A full e2e with real deposited notes means the nullifiers are the actual input note nullifiers, and the output commitments are freshly generated by the protocol (not random). |
| **Settler triggered from committee loop** | Currently the settler must be triggered manually or via the in-process loop. In production, the committee server must reliably be started with `DATABASE_URL` to activate the settler automatically. |
| **Multi-batch support** | Settler creates one job per match per batch. If a batch has N matches, N jobs are queued. The relayer handles each independently. Verify atomicity guarantees across multi-match batches. |

### mpc_settlement — what "build pending" means

The circuit source (`circuits/mpc_settlement/main.circom`) is complete and correct.
What still needs to happen before on-chain ZK verification works:

1. **Compile**: `bash circuits/mpc_settlement/build.sh` — generates `.r1cs`, `.wasm`, `.zkey`, `.vk.json` (takes 5-10 min, requires ~1.1 GB ptau download on first run).
2. **Deploy verifier**: run `circom2soroban vk circuits/mpc_settlement/output/main_verification_key.json` → deploy the Soroban verifier contract.
3. **Update `mpc_settle()`**: add `proof_bytes: BytesN<256>` + `pub_signals_bytes: BytesN<...>` args to the shielded pool contract and call the verifier.
4. **Redeploy pool**: contracts must be rebuilt after step 3.

The relayer worker (`MPC_SETTLE_SUBMIT`) already generates and passes proof bytes when artifacts are present (graceful fallback to committee-sig-only when they are not).

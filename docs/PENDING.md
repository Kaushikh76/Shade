# Shade Protocol — Pending

## Status as of 2026-07-01

All Phase 1 protocol flows are proven on real testnets (Arbitrum Sepolia + Stellar Testnet).
Phase 2 wallet architecture (Privy, note vault, user-signed flows, Next.js UI) is complete.
MPC committee matching with on-chain settlement is complete — settlement is proof-backed
(P0 #2/#3) and the 2-of-3 threshold requires distinct signers (P0 #1). **The "MPC" label
is aspirational, not descriptive of matching itself**: the coordinator decrypts and
reconstructs every intent's plaintext amount in one place to run matching (P2 #16 /
bible §14.2-14.3) — this is Shamir-share transport into a trusted matcher, not private
multi-party computation. Treat it as testnet-grade until the TEE matcher (V2, not
started) or real secure-comparison MPC (V3/V4) lands.
Phase A (RFQ↔MPC unified pipeline), Phase B (private amounts via Shamir + SHA-256 commitments),
and Phase C (ZK witness builder + proof generator + relayer integration) are complete.

See [testnet-transactions.md](testnet-transactions.md) for all real on-chain tx hashes.

fix has a `P0`/`P1`/`P2`/`P3`/`P4 #N` code comment at its call site

Remaining work is in three areas: **production hardening**, **new flows**, and **mainnet readiness**.

## MPC — Production Gaps (current implementation is testnet-grade)

| Item | Detail |
|---|---|
| **Persistent committee keypairs** | ✅ Fixed — `keys.ts` persists Ed25519 keypairs in `mpc_committee_keys` DB table via `loadOrGenerateKeys()`. |
| **Real Merkle root in `mpc_settle`** | ✅ Fixed — `computeMpcRoot` Windows bug resolved (removed `existsSync` guard that failed on `.exe` suffix). Real lean-IMT root now computed via `coinutils compute-root` and passed to `mpc_settle` (`cdf4abe`). |
| **Output commitments are `recipient_commitment`** | Current DB stores `recipient_commitment` as the output. The proper derivation: for each matched intent, the counterparty's `note_commitment` becomes the output commitment (i.e. intent A's output = intent B's input note, and vice versa). Review and tighten this mapping. |
| **Real intent amounts via Shamir** | Integration test uses random nullifiers/commitments. A full e2e with real deposited notes means the nullifiers are the actual input note nullifiers, and the output commitments are freshly generated by the protocol (not random). |
| **Settler triggered from committee loop** | Currently the settler must be triggered manually or via the in-process loop. In production, the committee server must reliably be started with `DATABASE_URL` to activate the settler automatically. |
| **Multi-batch support** | Settler creates one job per match per batch. If a batch has N matches, N jobs are queued. The relayer handles each independently. Verify atomicity guarantees across multi-match batches. |
| **Coordinator sees plaintext amounts (P2 #16)** | `coordinator.ts::runMatchingRound` decrypts every node's Shamir share and reconstructs the cleartext amount in-process to run matching, then immediately discards it (`entry.decryptedShare = null`) — but for that window, one process holds every intent's amount. Whether the 3 "nodes" run in one process or three, the *matcher* is fully trusted with all inputs today. This is Shamir-share transport + a trusted matcher, not privacy-preserving MPC. Staged fix: (1) **now** — call this out explicitly wherever "MPC" is mentioned in user/investor-facing docs, don't imply amounts are hidden from the matcher; (2) **V2** — move matching into a TEE (attested enclave) so no host operator sees plaintext (tracked below, not started); (3) **V3/V4** — real secure multi-party comparison, no single party ever reconstructs the amount. |
